建议将团队成员分为：**架构/基础设施（Ops）**、**核心引擎（AI/Logic）**、以及**数据/交付（Data/Dev）**。

---

## 👨‍💻 成员 A：架构与基础设施 (Cloud Native Engineer)

**核心目标：** 负责“跑得通”。解决所有环境、权限和流水线问题。

- **负责模块：**
    
    - **1. 触发层：** 部署 Serverless 函数，配置环境变量和 Cron 触发器。
        
    - **3. 存储层：** 搭建 OSS/S3 环境，编写通用的文件读写 SDK（带版本索引逻辑）。
        
    - **7. 可靠性：** 接入日志服务（SLS/CloudWatch）和告警推送（钉钉/邮件）。
        
- **交付物：** 一个稳定的函数运行环境，以及一套能让其他队友直接调用的 `storage_lib.py`。
    

## 🧠 成员 B：核心智能引擎 (AI/LLM Logic Engineer)

**核心目标：** 负责“想得深”。这是项目的灵魂，直接决定“增量对比”准不准
- **负责模块：**
    - **4. 增量对比：** 编写核心 Prompt，调优 LLM 提取数值变化的准确度。
    - **5. 冲突仲裁：** 编写仲裁逻辑脚本，实现权重评分系统。
    - **技术重点：** 负责将非结构化的文本转为结构化的 JSON 对比结果。
- **交付物：** `analyzer.py`（输入新旧文本，输出差异 JSON 和置信度）。

## 🛠️ 成员 C：数据采集与交付 (Data & Frontend Engineer)
**核心目标：** 负责“看得到”。确保有新鲜数据进来，且结果能清晰展示。
- **负责模块：**
    - **2. 采集层：** 编写爬虫或对接 API，实现去重逻辑和失败重试。
    - **6. 展示与报告：** 编写渲染逻辑，将差异 JSON 转为带高亮的 Markdown 报告。
    - **前端（可选）：** 如果有精力，可以用 Streamlit 或简单的 HTML 搭建一个仪表盘。
- **交付物：** `scraper.py`（数据源）和 `reporter.py`（可视化报告）。

---

## 🤝 协作路线图 (Milestones)
为了避免大家互相等待，建议按以下顺序对齐接口：
1. **定义数据契约（第 1 天）：** 三人共同商定 `Report` 的 JSON 格式（包含哪些字段，来源权重怎么写）。
2. **Mock 开发（第 2 天）：**
    - A 配置好云端 Hello World。
    - B 用手动复制的文本调优 Prompt。
    - C 找几个网页写爬虫 Demo。
3. **集成测试（第 3 天）：** A 将 B 和 C 的代码整合进 Serverless 函数，进行全链路跑通
